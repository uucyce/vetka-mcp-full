### 1. Что такое CAM (Constructivist Agentic Memory)

**Проблема**, которую решает CAM:
- LLM плохо справляются с long-form documents (>100k tokens).
- Нужна внешняя память, которая делает vanilla LLM **agentic** (автономным reading agent).
- Существующие memory (vector DB, graphs) — heuristic, без принципа.

**Решение**: Вдохновение от **Jean Piaget's Constructivist Theory** (теория когнитивного развития ребёнка).

**Три ключевых черты agentic memory**:
1. **Structured schemata** — иерархическая структура (tree).
2. **Flexible assimilation** — интеграция новой информации в существующую схему.
3. **Dynamic accommodation** — перестройка схемы под новую информацию (branching, pruning, merging).

**CAM** — прототип, реализующий эти три черты.

### 2. Архитектура CAM (формулы и алгоритмы)

#### 2.1 Memory Development (построение памяти)
- **Incremental overlapping clustering** для создания иерархии.
- Foundation layer: semantic network из raw chunks (text similarity + narrative coherence).
- Higher layers: hierarchical summarization.

**Формула clustering (упрощённо)**:
```python
# Similarity между chunks
sim(i,j) = cosine(embedding_i, embedding_j) + α * temporal_proximity(i,j)

# Overlapping clusters → tree structure
clusters = incremental_clustering(chunks, threshold=0.7, overlap=0.3)
```

#### 2.2 Memory Retrieval (Prune-and-Grow)
- **Associative exploration**: от query → activation релевантных узлов → расширение по соседям.
- **Prune-and-Grow strategy**: pruning нерелевантных веток, growing promising.

**Формула activation**:
```python
activation_score(node) = relevance(query, node) + β * connectivity(node)
# relevance = cosine(query_emb, node_summary_emb)
# connectivity = number of links to activated nodes
```

#### 2.3 Dynamic Operations
- **Branching** — новая ветка для альтернатив/обновлений.
- **Pruning** — удаление low-relevance.
- **Merging** — слияние похожих поддеревьев.

### 4. Применение к VETKA (Phase 17)

**Ты прав на 200%** — CAM **идеально ложится** на поведение VETKA при добавлении новой информации:

| CAM механизм          | VETKA аналог                          | Формула/код пример |
|-----------------------|---------------------------------------|-------------------|
| Branching (новая ветка) | Promote новой ветки при новом артефакте | `if new_info_entropy > threshold: create_branch()` |
| Pruning               | Удаление low-entropy/dup веток (Z-ось) | `prune if activation_score < 0.3` |
| Merging               | Слияние похожих поддеревьев (>0.92 sim) | `merge if cosine(subtree_emb_a, subtree_emb_b) > 0.92` |
| Accommodation         | Dynamic subtree height + repulsion     | Твоя текущая формула! |
| Assimilation          | Incremental layout update             | Soft force relaxation |

**Поведение при добавлении**:
- Новая инфа → **branching** (новая ветка).
- Если конфликтует → **accommodation** (перестройка: repulsion, height recalc).
- Если дубликат → **pruning/merging**.

Это делает VETKA **конструктивистской памятью** — не статичным графом, а живой системой, которая эволюционирует.

# Основной цикл CAM (упрощённо из repo)
def cam_process(document_chunks):
    # Development: build hierarchy
    memory_tree = incremental_clustering(chunks)

    # Branching for new info
