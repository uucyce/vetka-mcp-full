## 2. **Подводные камни (Pitfalls) — Что не учли?**
Из похожих setups (LangGraph/AutoGen/Prefect) + твоего doc:

| Pitfall | Risk in Your Arch | Fix/Suggestion |
|---------|-------------------|---------------|
| **Race Conditions (JSON Files)** | Multi-write `task_board.json` (MYCELIUM writes, VETKA reads). OS FS ok local, но concurrent pipelines → corruption. | ✅ Уже: Append-only + atomic writes (json.dump atomic via temp file). Add `filelock` lib. Phase 3: `flock` или Redis only if scale. |
| **Qdrant Contention** | Both poll `localhost:6333` (reads ok), но heavy indexing (VETKA Watcher) → spikes. | Low risk (HTTP stateless). Monitor: `vetka_get_metrics`. Pitfall from LangChain: Add Qdrant clustering (future Phase 4). |
| **Process Crash Isolation** | MYCELIUM crash → stale tasks (ok, doc says "cancelled"). VETKA crash → DevPanel lives (direct WS). | ✅ Good. Add: PID watchdog (heartbeat checks both). Prefect-style: `crash_recovery.json` per task. |
| **LLM Latency/Timeouts** | MYCELIUM async good, но Cursor MCP stdio timeout (default 120s) на long pipelines. | Tune `.mcp.json` env: `MCP_TIMEOUT=600`. Pitfall: Key exhaustion (history: [ERROR] OpenRouter keys). → UnifiedKeyManager rotate + fallback. |
| **Scale (Multi-MYCELIUM)** | 1 pipeline now. Parallel? Files → locks. | Phase 4: Prefix files `task_board_worker1.json`. Or Dask for orchestration. |
| **Security** | Localhost only, но remote (Phase 4) → expose 8082? | HTTPS + auth tokens in WS. |
| **Memory Leaks** | STM per-pipeline in MYCELIUM RAM → OOM on 10+ pipelines. | TTL evict (e.g., 1h post-done). Qdrant for LTM dump. |
| **DevPanel UX** | Direct WS fast, но if MYCELIUM down → blank. | Fallback: Poll VETKA `/task-board`. |

**Общий вердикт**: 95% robust. Главный gem — **no new infra**. Pitfalls mostly scale-related (Phase 4).

## 3. **STM в MYCELIUM — Правильно ли? Нужен ли в VETKA?**
**Да, правильно! Ты прав в архитектуре, но уточню:**

- **Почему в MYCELIUM only** (из doc):
  ```
  STM (short-term memory): per-pipeline, lives in MYCELIUM process only
  ```
  - VETKA: **Stateless/fast** (<100ms tools: search/files/UI). Нет pipelines/agents — pure proxy (REST + lightweight). Нет нужды в per-task memory (conversation context via `vetka_get_conversation_context` из Qdrant).
  - MYCELIUM: **Stateful pipelines** (Scout→Architect→... 1-15min). STM = pipeline-local (e.g., researcher passes facts to coder). Не shared, чтоб no contention.

- **Тебе не нужен STM в VETKA**:
  - VETKA agents (если есть) — simple (search, git). Используют **shared LTM**: Engram/CAM/KG из Qdrant.
  - Если добавишь agents в VETKA (e.g., quick chat replies) — используй **session-scope STM** via Qdrant collections (per-chat_id). Не process-local.
  - Pitfall if migrate: Shared STM → race conditions in event loop.

**Вектора подтверждают** (pinned doc: "Memory: Shared Data, Separate Routing"):
```
Qdrant shared → VETKA (Knowledge Mode) ╲ MYCELIUM (context injection + STM routing)
STM per-pipeline → MYCELIUM only
```
Viewport files (MYCELIUM_TASKS.md, PHASE_128_ASSIGNMENTS.md) фокусированы на pipelines → MYCELIUM owns.

**Предложение**: Add `vetka_stm_lite` tool? Но no — держи split clean. Если VETKA agents вырастут → Phase 3: Shared Redis for cross-process STM.
