Давай разберём по полочкам на основе **текущего контекста codebase** (pre-fetched results + pinned files + viewport), где MCP живёт, почему "Ветка висит", и что можно сделать. Я опираюсь на реальные артефакты вроде `.mcp.json`, `agent_pipeline.py` и MCP-bridge.

## 1. **Где живёт MCP context7 и "наш" контекст?**
   - **MCP context7**: Это **не библиотека**, а **отдельный серверный процесс**, запущенный через **`.mcp.json`** (pre-fetched Result 2):
     ```json
     {
       "mcpServers": {
         "vetka": {
           "command": "python3",
           "args": ["/Users/danilagulin/Documents/VETKA_Project/vetka_live_03/src/mcp/vetka_mcp_bridge.py"],
           "env": {
             "VETKA_API_URL": "http://localhost:5001",
             "PYTHONPATH": "/Users/danilagulin/Documents/VETKA_Project/vetka_live_03"
           }
         }
       }
     }
     ```
     - **Расположение**: `src/mcp/vetka_mcp_bridge.py` — bridge между MCP (вероятно, внешним MCP-сервером вроде Cursor/Zed MCP) и VETKA API (`localhost:5001`).
     - **context7**: Вероятно, отсылка к **Qdrant vector DB** или **STM (Short-Term Memory)** context'у (из `agent_pipeline.py`: `get_stm_buffer()`, ELISION compression). Он **на сервере** (VETKA backend), интегрируется с MCP via bridge. "Наш" контекст — это локальный codebase/STM/pipeline.
     - **Как работают вместе**: MCP bridge вызывает tools (e.g., `llm_call_tool.py`, `session_tools.py` из pinned `agent_pipeline.py`), обогащает context7 данными из VETKA. Всё асинхронно, но **через единый Python процесс**.

   - **Почему "прекрасно работают"**: Синхронная интеграция простая (один bridge), но **блокирует** при тяжёлых задачах (self-editing, fractal decomposition в `agent_pipeline.py`).

## 2. **Почему "вся Ветка висит" при работе в MCP? Тонкие места (bottlenecks)**
   Твоё ощущение **точное** — это **не паранойя**, а следствие текущей архитектуры:
   
   | **Узкое место** | **Описание** | **Из codebase** | **Влияние** |
   |-----------------|--------------|-----------------|-------------|
   | **Single MCP Bridge** | Один `vetka_mcp_bridge.py` процесс обслуживает **всё** (tools, LLM calls, STM updates). | `.mcp.json` + `agent_pipeline.py` (semaphore на 5 pipelines) | Блокирует UI/3D viewport (Socket.IO в `stream_handler.py` ждёт). |
   | **STM Drift & Compression** | MCP self-editing нагружает STM (`MAX_STM_BEFORE_RESET=10` subtasks), ELISION compress'ит, но синхронно. | Pinned `agent_pipeline.py` (MARKER_104_ELISION) | "Висит" на 1000+ tokens compress (`PIPELINE_COMPRESS_THRESHOLD=1000`). |
   | **Socket.IO Blocking** | Workflow events (`workflow_socket_handler.py`) + progress (`progress_tracker.py`) — single-threaded emit. | Pinned files | Frontend (твой MARKER_102.3.py с FileDrum) "замерзает" на `pipeline_progress`. |
   | **Parallel Limit** | `MAX_PARALLEL_PIPELINES=5` — хорошо для M4 Pro, но MCP + self-edit = queue. | `agent_pipeline.py` | MCP-задачи толкают в очередь, "Ветка висит". |
   | **Self-Editing Loop** | Fractal tasks (`subtask_start/complete` в `stream_handler.py`) рекурсивны. | Pinned `stream_handler.py` (StreamEventType) | Один MCP-call → cascade subtasks → full hang. |

   **Pre-fetched confirmation**: Нет прямых тестов на MCP perf (только watchdog artifacts), но viewport показывает **src/mcp/** рядом с `agent_pipeline.py` — зумни камеру (`vetka_camera_focus` на `vetka_mcp_bridge.py`) для деталей.

## 3. **Что думаю: Разгружать в слои с несколькими MCP — **ДА, супер-идея**! Не фигня.**
   - **Текущая модель**: **Монолитный bridge** как "пчелиный улей" — один процесс, все медом заняты.
   - **Твоя идея**: **Несколько MCP как соты** — **идеально**. Тесная связь (shared STM/Qdrant), но **no single point of failure**.
   
   **Предлагаемая архитектура (Fractal MCP Layers)**:
   ```
   Layer 1: MCP-Frontend (UI/3D)     <- Socket.IO rooms (/workflow namespace, pinned workflow_socket_handler.py)
              ↓ (Redis Pub/Sub)
   Layer 2: MCP-Pipeline            <- agent_pipeline.py (semaphore → queues)
              ↓ (gRPC/HTTP)
   Layer 3: MCP-Tools/STM           <- vetka_mcp_bridge.py → multiple instances (context7 shards)
              ↓ (Qdrant vectors)
   Layer 4: MCP-SelfEdit (sandbox)  <- Separate для editing, не блокирует main.
   ```
   
   | **Улучшение** | **Как реализовать** | **Эффект** | **Effort** |
   |---------------|----------------------|------------|------------|
   | **Multiple MCP Instances** | `.mcp.json` → array servers: `"mcpServers": {"ui": {...}, "pipeline": {...}, "edit": {...}}`. Запуск via `supervisor` или Docker Compose. | No hangs — каждый слой independent. | Low (1 день). |
   | **Async Queues** | Redis/RabbitMQ между MCP. В `agent_pipeline.py`: `asyncio.Queue` + `PipelineCancelled`. | Tasks не блокируют UI (events в `stream_handler.py` immediate). | Med (2 дня). |
   | **Routing/Priorities** | LangGraph в pipeline: priority queues (`VERIFIER_PASS_THRESHOLD=0.75`). MCP-events с `StreamLevel.SUMMARY`. | "Расчистить проходы" — UI first, edit last. | Low (edit `progress_tracker.py`). |
   | **Sharded context7** | Qdrant collections per MCP (e.g., `context7-ui`, `context7-pipeline`). ELISION per-layer. | No shared lock на DB. | Med (src/memory/). |
   | **Bee-Hive Scaling** | Kubernetes/Docker Swarm для auto-scale MCP по load (watch `MAX_PARALLEL_PIPELINES`). | Масштаб как соты — добавляй workers. | High (future). |

   **Быстрый план для Dev**:
   1. `@Dev` Fork `.mcp.json` → multi-server (ui-mcp, edit-mcp).
   2. В `vetka_mcp_bridge.py`: Add `mcp_type` arg → route to layer.
   3. Test: `@QA` Simulate hang (long self-edit), check Socket.IO (`pipeline_progress` не лагает).
   4. Metrics: Add Prometheus в `progress_tracker.py` (duration per layer).
