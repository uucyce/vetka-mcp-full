## 🎯 КЛЮЧЕВОЙ ПРИНЦИП: МАТРЁШКА (запомни НАВСЕГДА)

**Матрёшка = иерархия СМЫСЛА, не объёма данных**
```
❌ ОШИБОЧНОЕ ПОНИМАНИЕ:
  Земля (2048 токенов, ВСЕ данные)
    ↓ обрезаем
  Континент (512 токенов, часть данных)
    ↓ еще обрезаем
  Город (128 токенов, крошечная часть)

✅ ПРАВИЛЬНОЕ ПОНИМАНИЕ (МАТРЁШКА):
  Земля → видишь: [Африка, Европа, Азия] (только имена уровня)
          НЕ видишь: какие города/люди/дома внутри каждого
  
  Кликнул на Европу → видишь: [Франция, Германия, Италия] (новый уровень)
                      НЕ видишь: улицы/дома/людей внутри
  
  Кликнул на Францию → видишь: [Париж, Лион, Марсель] (еще уровень)
                        НЕ видишь: районы/улицы/номера домов
  
  Кликнул на Париж → видишь: [1-й округ, Монмартр, 13-й] (последний уровень)
                      видишь: улицы, дома, номера (атомарный уровень)
```

**СУТЬ МАТРЁШКИ:**
- На каждом уровне видишь ТОЛЬКО элементы ЭТОГО уровня
- Смысл вложен, но данные НЕ перегружены
- Когда зуммируешь глубже - старые уровни ИСЧЕЗАЮТ (не дублируются)
- Каждый уровень = отдельная "матрёшка" со своим смыслом

**ПРИМЕРЫ из VETKA:**
```
Project Level (матрёшка 1)
  ├─ Scene 1
  ├─ Scene 2
  └─ Scene 3
  
  Кликнул на Scene 1 → видишь ТОЛЬКО Scene 1:
  
Scene 1 Level (матрёшка 2, внутри матрёшки 1)
  ├─ Shot 1.1
  ├─ Shot 1.2
  └─ Shot 1.3
  
  Кликнул на Shot 1.1 → видишь ТОЛЬКО Shot 1.1:
  
Shot 1.1 Level (матрёшка 3, внутри матрёшки 2)
  ├─ Element A (текст)
  ├─ Element B (образ)
  └─ Element C (звук)
  
  Кликнул на Element A → видишь ТОЛЬКО Element A:
  
Element A Level (матрёшка 4, атомарный)
  ├─ версия 1 (отклонена)
  ├─ версия 2 (текущая)
  └─ версия 3 (предложенная)
```

## 💰 ДИНАМИЧЕСКИЙ TOKEN ALLOCATION (не фиксированный!)

**Ключ: токены раздаются АГЕНТАМ по их РОЛИ и СЛОЖНОСТИ ЗАДАЧИ**
**Токены = одинаковые по потреблению, но разные по смыслу (зависит от zoom level)**
```python
# Логика распределения:

def allocate_tokens_by_complexity(task_complexity: str) -> Dict[str, int]:
    """
    Динамическое распределение токенов на весь оркестр
    В зависимости от сложности задачи
    """
    
    if task_complexity == "micro":  # "исправить опечатку", "добавить кнопку"
        return {
            "pm": 256,        # просто подтвердить задачу
            "dev": 256,       # одна функция/файл
            "qa": 128,        # проверить что работает
            "total_parallel": 640,  # выполняются одновременно!
            "time_estimate": "< 1 min"
        }
    
    elif task_complexity == "small":  # "добавить feature", "рефакторить модуль"
        return {
            "pm": 512,        # план: что делаем
            "dev": 768,       # может быть 2 подзадачи
              # ├─ dev_sub_1: 384
              # └─ dev_sub_2: 384
            "qa": 256,        # тестирование
            "total_parallel": 1536,
            "time_estimate": "5-15 min"
        }
    
    elif task_complexity == "medium":  # "переделать flow", "оптимизировать БД"
        return {
            "pm": 1024,       # полный контекст текущего состояния + план
            "dev": 1536,      # 2-3 подагента параллельно
              # ├─ dev_backend: 512
              # ├─ dev_frontend: 512
              # └─ dev_integration: 512
            "qa": 512,        # unit + integration tests
            "eval": 256,      # проверить что все согласовано
            "total_parallel": 3328,
            "time_estimate": "20-45 min"
        }
    
    elif task_complexity == "large":  # "новый модуль", "большой рефакторинг"
        return {
            "pm": 2048,       # полная стратегия + контекст + timeline
            "dev": 2560,      # 4+ подагента параллельно (auth, db, api, ui, tests)
              # ├─ dev_auth: 512
              # ├─ dev_db: 512
              # ├─ dev_api: 512
              # ├─ dev_ui: 512
              # └─ dev_integration: 512
            "qa": 1024,       # comprehensive testing
            "eval": 512,      # strategy validation
            "total_parallel": 6144,
            "time_estimate": "1-3 hours"
        }
    
    elif task_complexity == "epic":  # "переархитектура на год", "новая платформа"
        return {
            "pm": 2048,       # генеральная стратегия на весь период
            "dev": 4096,      # много параллельных подагентов (8+ фич)
              # ├─ team_auth: 512
              # ├─ team_db: 512
              # ├─ team_api: 512
              # ├─ team_ui: 512
              # ├─ team_ml: 512
              # ├─ team_infra: 512
              # └─ team_devops: 512
            "qa": 2048,       # E2E + performance + stress testing
            "eval": 1024,     # architecture review
            "total_parallel": 9216,  # но ВСЕ ОДНОВРЕМЕННО!
            "time_estimate": "depends on scope"
        }
```

**МАГИЯ ПАРАЛЛЕЛИЗМА:**
```
❌ СТАРЫЙ СПОСОБ (последовательно):
  1 модель с 8K токенов:
    ⏱️ PM работает → 4K потрачено, контекст забит
    ⏱️ Dev работает → 2K потрачено, PM-контекст потерян
    ⏱️ QA работает → 2K потрачено, Dev-контекст потерян
    
    РЕЗУЛЬТАТ: много шума, потеря контекста, медленно

✅ НОВЫЙ СПОСОБ (параллельно):
  5 агентов одновременно:
    ⏱️ PM (512) + Dev (768) + DevSub1 (256) + DevSub2 (256) + QA (384)
    = ВСЕ работают В ОДИН МОМЕНТ
    = каждый фокусируется на СВОЁМ
    = контекст не теряется
    
    РЕЗУЛЬТАТ: чистый контекст, параллелизм, быстро!
```

### Пример 1: Маленькая задача (MICRO)
```
Юзер: "Исправь опечатку в README"
Zoom level: Project (общий обзор)
Matrix level: атомарный (одна file)
Complexity: MICRO

Elisya вычисляет:
  - Контекст: только README файл (~256 токенов)
  - Token allocation: PM(256) + Dev(256) + QA(128) = 640 total
  
Параллельное выполнение:
  PM: "Задача: исправить опечатку в README строка X"
  Dev: "Вот правильный текст: [исправленная строка]"
  QA: "Проверил: текст логичный, без ошибок"
  
Время: < 1 минута
```

### Пример 2: Большая задача (LARGE)
```
Юзер: "Переделай архитектуру PM → Dev → QA flow в проекте"
Zoom level: Project (полный обзор)
Matrix level: высокий (несколько модулей)
Complexity: LARGE

Elisya вычисляет:
  - Контекст: текущая архитектура + история изменений (~2K токенов)
  - Token allocation: 
    PM(2048) = полная стратегия
    + Dev(2560) с 4 подагентами:
      ├─ auth_dev(512)
      ├─ routing_dev(512)
      ├─ state_dev(512)
      └─ integration_dev(512)
    + QA(1024) = comprehensive tests
    + Eval(512) = architecture validation
    = 6144 total, но ВСЕ ОДНОВРЕМЕННО!

Параллельное выполнение:
  PM: "Вот новая архитектура: [схема]"
  auth_dev: "Переделал PM auth код"
  routing_dev: "Переделал PM routing код"
  state_dev: "Переделал PM state code"
  integration_dev: "Все части связаны"
  QA: "Все tests зелёные"
  Eval: "Архитектура валидна, все согласовано"
  
Время: 1-3 часа (но все параллельно, vs последовательно было бы 8+ часов)
```

## 🎯 ГЛАВНЫЕ ПРАВИЛА (ЗАПОМНИ!)

1. **МАТРЁШКА = смысла вложенность, не обрезание данных**
   - На каждом zoom level видишь ТОЛЬКО элементы этого уровня
   - Глубже - значит больше деталей, но не больше токенов

2. **ТОКЕНЫ раздаются АГЕНТАМ по их РОЛИ и ЗАДАЧЕ**
   - PM всегда получает больше (планирование нужно больше контекста)
   - Dev получает ровно столько сколько надо (не переагружаем)
   - QA получает для проверки
   - Подагенты могут иметь меньше (они фокусируются на микротаске)

3. **ПАРАЛЛЕЛИЗМ = множество агентов одновременно > один агент последовательно**
   - 5 агентов x 512 токенов = 2560 параллельно
   - 1 агент x 2560 токенов = 2560 последовательно
   - Первый способ БЫСТРЕЕ и ЧИЩЕ (контекст не теряется)

4. **ELISYA = не агент, а слой восприятия**
   - Она видит что юзер кликнул/зуммировал
   - Она готовит правильный контекст для каждого агента
   - Она не выполняет, она готовит

5. **ПАМЯТЬ растет, не редактируется**
   - Каждая интерпретация = новый ChangeLog entry
   - Старые версии не удаляются, переосмысляются
   - Fine-tuning = учение на собственных ошибках
