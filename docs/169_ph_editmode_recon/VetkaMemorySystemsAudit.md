# Vetka Memory & Context Tools Audit

## 1. Layered memory stack
- **CAM (Context-Aware Memory):** `src/orchestration/cam_engine.py` tracks context signals (node heat, user prefs) and powers `camScore` injection (see `docs/119_ph_Opus/PHASE_119_2_CAM_LABELS_RECON.md`). CAM emits `context_scores` used by agents and the UI (e.g., artifact panel highlights). As shown in `docs/besedii_agents_o_vetke.txt`, CAM remembers user preferences while Elysia chooses tools, so they work together (CAM stays fast, Elysia handles tool logic).
- **ENGRAM (Long Memory):** `src/memory/engram_user_memory.py` keeps persistent embeddings, organizes them via MGC cascades, and feeds the unified recall pipeline described in `docs/155_ph/MARKER_155_MEMORY_DEPENDENCY_RECON_AND_SHARED_POLICY.md`. Engram entries propagate through `agent_memory_hook`, `context_fusion`, and the `Myco` stack (see `docs/99_ph_memory_architecture/PHASE_99_UNIFIED_PROMPT.md`). Engram serves as the reliable long-term index (hotâ†’warmâ†’cold via `MGCCache`).
- **STM (Short-Term Memory):** `src/memory/stm_buffer.py` caches recent message contexts before they age into Engram/MGC. `docs/118_ph/STM_UNIFICATION_PLAN.md` explains how STM eviction adds entries into MGC when capacity hits thresholds, ensuring the system falls back to still-relevant threads.
- **MGC (Multi-Generational Cache):** Implemented in `src/memory/mgc_cache.py` (Phase 99). Three generations: Gen0 RAM (hot, limited to `gen0_max`), Gen1 Qdrant (warm), Gen2 JSON (cold), with optional archive/backfill (Phase 99, 118). Tests `tests/test_mgc_cache.py` and `tests/test_phase99_integration.py` verify promotions and fallback behaviors. MGC is referenced in `user_message_handler` for prefetch weighting (`MARKER_114.8`). `docs/99_ph_memory_architecture/QUICK_START_PHASE_99.md` describes pipeline `Request â†’ MemoryProxy â†’ MGC â†’ Response`.
- **ELISION + HOPE:** `docs/154_ph/MARKER_155_AUDIT_REPORT.md` and `src/memory/spiral_context_generator.py` show how Elision compresses context and HOPE adds prioritization, both streaming into CAM/STM/MGC to limit noise. The `SpiralContextGenerator` uses MGC entries when generating contexts for retrieval tasks.
- **Elysia tools:** `src/orchestration/elysia_tools.py` (Phase 75) provides Weaviate-based tool selection for code operations, with direct bypass class `ElysiaToolsDirect`. Docs `docs/104_ph/SCAN_ELYSIA.md` and `docs/73_ph/ELISYA_INTEGRATION_AUDIT.md` explain the decision tree, fallback when `elysia-ai` missing, and how CAM preferences interplay with Elysia's selection. Elysia acts as context-aware tool navigator, referencing CAM/ENGRAM signals to decide which tool to run.

## 2. Connecting to the memory core & sandbox copying
- **How to hook in:** import `get_mgc_cache()`, `STMBuffer`, `EngramUserMemory` and connect them through `context_fusion` or message handlers (e.g., `src/api/handlers/user_message_handler.py`). To observe CAM, track `cam_engine` events and inspect `CAM` labels on files (`docs/119_ph_Opus`). When instrumenting Elysia, call `elysia_tools.ElysiaTree` to let it pick the best tool; use `ElysiaToolsDirect` for low latency paths. The `docs/155_ph/...` memory dependency recon ensures you follow `STM/ELISION/MGC/CAM` flows.
- **Safe sandbox copy:** replicate `src/memory` and `src/orchestration/elysia_tools.py` to sandbox directory (e.g., `rsync -av src/memory ~/sandbox/src/memory`). Copy `docs/118_ph/MEMORY_DEDUP_PLAN.md` and `docs/99_ph_memory_architecture` for reference. Avoid shipping real embeddings by wiping Engram/Qdrant data before sandboxing; instead, supply synthetic JSON in `tests/test_phase99_integration.py`. When experimenting with Elysia, set `ELYSIA_ENABLED=false` or install stub dependencies to prevent hitting live Weaviate.
