# Vetka Scan & Search Core Audit

## 1. Flow of scanning & vectorization
- **Entry points:** `src/api/routes/watcher_routes.py` handles `POST /api/watcher/add`, `/add-from-browser`, `/index-file`, plus `GET /status/heat`. These routes register folders/files and invoke `LocalScanner`, `FileWatcher`, and `EmbeddingPipeline`. See markers `MARKER_158.WATCHER.*` for coverage. `docs/158_ph/PHASE_158_MULTIMEDIA_QDRANT_MARKER_RECON_2026-03-02.md` lists statuses and remaining gaps for browser metadata, binary placeholders, and OCR reachability.
- **Supported extensions:** `file_watcher.SUPPORTED_EXTENSIONS` and `LocalScanner.SUPPORTED_EXTENSIONS` already include pdf/img/audio/video. Regression test `tests/test_phase158_watcher_multimedia_extensions.py` ensures the scanner handles cur media types (`MARKER_158.SCANNER.EXT_WATCHER_ADD_MEDIA`).
- **Extractor registry / pipeline:** `src/scanners/extractor_registry.py` (marker `MARKER_158.INGEST.MEDIA_EXTRACTOR_REGISTRY`) unifies OCR/STT/Pulse/JEPA hooks; triple-write and Watcher index call it. `src/scanners/multimodal_contracts.py` defines `media_chunks_v1` schema used by Qdrant payload (`MARKER_158.QDRANT.SCHEMA_MEDIA_CHUNKS_V1`). `docs/158_ph/VETKA_UNIFIED_IO_MATRIX_V1.md` enumerates canonical JSON contract fields (`scene_node_id`, `timeline_lane`, `rhythm_features`, `cam_features`).
- **Vectorization & storage:** `src/scanners/qdrant_updater.py` + `qdrant_batch_manager.py` write to Qdrant; they include `markes` for incremental update and media chunk schema. `TripleWrite` ingestion saves vector payloads, while tests ensure `media_chunks` retrieval for preview/semantic links (`MARKER_158.QDRANT.SCHEMA_MEDIA_CHUNKS_V1`). Input matrix documented in `docs/158_ph/VETKA_UNIFIED_IO_MATRIX_V1.md` and `docs/158_ph/PHASE_158_MULTIMEDIA_VJEPA_ROADMAP_CHECKLIST_2026-03-02.md` covering ingest/extract/retrieve/render steps.
- **Semantic augmentation:** `POST /api/artifacts/media/semantic-links` uses TripleWrite search to find linked segments (hero/action/location/theme) and feeds lane IDs/take IDs. `media/rhythm-assist`, `media/cam-overlay` consume `media_chunks_v1` plus heuristics. Documented in `VETKA_MEDIA_MODE_FOLDER_SPEC_V1` and Phase 158 roadmap.

## 2. How to connect & sandbox copy
- **Core modules:** The scanning stack is rooted in `src/api/routes/watcher_routes.py`, `src/scanners/extractor_registry.py`, `src/scanners/multimodal_contracts.py`, and `src/scanners/qdrant_updater.py`. To inspect or extend, open those files—no compile step required (Python/FastAPI). For sandbox testing, copy the relevant directories into a safe runtime (`cp -r src/api routes` etc) or use `venv`? Use `rsync --archive --exclude='*.pyc' ...` to clone onto a sandbox machine, then run `python -m pytest tests/test_phase158...`.
- **Data handling:** Vector payloads live in Qdrant collections; to sandbox, run a local `qdrant` instance (see `docs/158_ph/VETKA_MEDIA_PIPELINE_BUDGETS_V1.md` for telemetry/route budgets). Use `docker compose up qdrant` and configure `src/config/qdrant` endpoints. Avoid pointing tests to production to prevent data leaks.
- **Markers for reference:** Use `MARKER_158.GAP.F2/F3/F5/F6` to identify incomplete browser import/OCR/triple-write flows before copying or modifying; they indicate areas that still need safety checks.

## 3. Recommendations
1. Capture extractor inputs/outputs within sandbox by mocking `watcher_routes` calls (send `req.multimodal=true`). 2. Validate `media_chunks_v1` payload in sandbox Qdrant before exposing to UI (`media_preview`, `semantic_links`). 3. Keep telemetry fields (`route_mode`, `degraded_mode`, `latency_ms`) from budgets doc to monitor ingestion health in staging.
