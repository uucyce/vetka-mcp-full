MARKER_94.8: CRITICAL MODEL ROUTING BUG - INVESTIGATION COMPLETE
================================================================================

SUMMARY
-------
A critical bug was discovered and FIXED where model routing was selecting the
wrong provider for OpenRouter's xAI models, causing 403 errors.

CRITICAL FINDINGS
-----------------

1. BUG LOCATION
   File: src/elisya/provider_registry.py
   Function: ProviderRegistry.detect_provider(model_name: str)
   Lines: 978-1043 (after fix)

   Marker: MARKER_94.8_BUG_ROUTING

2. THE BUG
   Problem: When user calls `x-ai/grok-4` (OpenRouter format)
   System was routing to: XAI direct API (api.x.ai/v1/chat/completions)
   Expected: OpenRouter API (openrouter.ai/api/v1/chat/completions)
   Result: HTTP 403 Forbidden error

   Root cause: Model ID prefix `x-ai/` was incorrectly interpreted as
   "direct xAI API indicator" instead of "OpenRouter xAI model format"

3. THE FIX
   Fixed the detect_provider() logic to check:
   - Direct API format FIRST (grok-* with no slash)
   - Then provider-prefixed format (x-ai/* with slash)

   This matches the OpenRouter convention where `provider/model` format
   always means "use OpenRouter, not direct API"

BEFORE (Lines 1005-1011):
    elif (
        model_lower.startswith("xai/")

        return Provider.XAI  # ❌ WRONG for x-ai/ format

AFTER (Lines 1006-1022):

    if model_lower.startswith("grok-") or model_lower == "grok":
        return Provider.XAI  # grok-4, grok-beta (direct API)

    # Then provider-prefixed format (OpenRouter)
    elif model_lower.startswith("xai/") or model_lower.startswith("x-ai/"):
        return Provider.OPENROUTER  # x-ai/grok-4, xai/grok-4 (OpenRouter)

CONCLUSION
================================================================================

CRITICAL BUG: ✅ FOUND AND FIXED

The model routing bug was caused by incorrect interpretation of model ID
prefixes. The system was treating `x-ai/` as a "direct API indicator" when
it actually represents the OpenRouter format for xAI models.

The fix implements the correct logic:
  - Direct model ID (no slash) → Use direct provider API
  - Provider/model ID (with slash) → Use OpenRouter

This enables users to successfully use OpenRouter's xAI models while
maintaining backward compatibility with direct API calls and all other
providers.
