## Recon Findings

### MARKER_157_6_RECON_F1: Скорость первого ответа уже конкурентная
- `first_response_ms`: ~1.4-3.4s (живой диалог возможен).
- Главный источник "тяжести" не старт, а длинный turn-close.

### MARKER_157_6_RECON_F2: Длинные хвосты в turn runtime
- Наблюдалось `total_turn_ms` до ~25-26s.
- Причина: `chunk3_smart/chunk4_deep` иногда тянут полный цикл, хотя draft уже есть.
- Вывод: для разговорного UX stage3/4 должны быть bounded-by-time и/или decoupled follow-up.

### MARKER_157_6_RECON_F3: STT нестабилен, не TTS
- `stt_confidence` проседает до 0.2-0.3, появляется `stt_retry`.
- В такие моменты TTS не главный bottleneck, первичный фейл в распознавании.
- Вывод: нужен приоритет надежного transcript_hint и мягкий порог retry.

### MARKER_157_6_RECON_F4: Языковой drift still exists
- При `stt_lang=ru` встречался `response_lang=en`.
- В коде есть memory-driven language hints, но stage refinement может уходить в EN.
- Вывод: language policy нужно централизовать на turn-level (одно решение языка на ход).

### MARKER_157_6_RECON_F5: ENGRAM contract критичен и должен быть source-of-truth
- Пользовательское требование: без bypass ENGRAM.
- Схема ключей уже расширена (`preferred_language`, `last_assistant_language`, `user_name`).
- Вывод: любые identity/language decisions только через ENGRAM + STM/CAM context.

### MARKER_157_6_RECON_F6: Контекст стал легче, но качество still generic
- `context_chars` теперь заметно ниже (порядок ~7-8k), это хорошо.
- Но ответы остаются общими в части "что вижу".
- Вывод: проблема не только в объеме, но и в приоритизации viewport facts в prompt assembly.

### MARKER_157_6_RECON_F7: Voice continuity must be fixed as product invariant
- UX-требование: один голос VETKA в сессии.
- Нужен hard contract в trace: `voice_profile`, `tts_engine`, `filler_audio_emitted`.

## eSpeak-ng Feasibility (C-3PO branch)

### MARKER_157_6_RECON_FS1: Технически feasible
- `espeak-ng` подходит для ultra-fast local TTS (CPU-only).
- Легко встраивается как subprocess provider в `tts_engine.py`.
- Может работать как:
  1. `force provider` для robo-mode,
  2. `fallback` при Edge 503/timeout.

### MARKER_157_6_RECON_FS2: Что проверяем в первую очередь
- RU intelligibility (не только скорость).
- Single-voice continuity.
- Fallback smoothness (без тишины/нойза).

---

## Risks
1. `espeak-ng` RU quality может быть слишком грубой.
- Mitigation: Edge default, eSpeak optional/fallback.

2. Stage3/4 latency может нивелировать выигрыш TTS.
- Mitigation: strict stage timeout + quick-close policy.

3. Сильный STT шум маскирует реальный прирост TTS.
- Mitigation: STT stabilization pass before final A/B claims.
