## Одно предложение

Mycelium автоматически строит карту проекта, расставляет задачи, подбирает команды, выполняет работу — пользователь только подтверждает.

---

## Проблема сейчас

Открываешь MCC — видишь 20+ кнопок, 3 тулбара, фильтры на пол-экрана. Непонятно что нажать первым. Непонятно что с чем связано. Создатель системы не понимает как ей пользоваться — значит система сломана.

---

## Принцип

**Максимум 3 действия за раз.** Не 3 кнопки плюс навигация плюс настройки. А 3 действия — всё что видит пользователь как выбор. Остальное скрыто или происходит автоматически.

---

## Матрёшка: 4 уровня, один экран

Mycelium — это DAG-в-DAG. Четыре уровня вложенности, один и тот же экран, разный масштаб:

```
Уровень 0: ROADMAP    — граф задач проекта (корень внизу, новое вверху)
Уровень 1: ЗАДАЧА     — клик на ноду, внутри — workflow DAG
Уровень 2: ИСПОЛНЕНИЕ — workflow запущен, ноды светятся по мере работы
Уровень 3: РЕЗУЛЬТАТ  — работа завершена, код/diff/лог
```

Переход между уровнями: клик на ноду = вход внутрь, кнопка "назад" = выход наружу. Как в файловом менеджере. Не новые панели — тот же экран, другое содержимое.

---

## Уровень 0: Roadmap

**Что видит пользователь:**
- Граф. Ноды = задачи. Стрелки = зависимости. Корень внизу.
- Архитектор уже расставил задачи и рекомендует следующую.
- Подсветка: что готово (зелёное), что следующее (яркое), что заблокировано (тусклое).

**3 действия:**
1. **Запустить** (рекомендованную задачу)
2. **Спросить Архитектора** (открыть чат)
3. **Добавить задачу** (ручной ввод)

**Откуда берётся Roadmap:**
- Архитектор анализирует кодовую базу и строит граф автоматически.
- Пользователь может поправить (перетащить, удалить, добавить).
- Roadmap сохраняется, обновляется по мере выполнения задач.

---

## Уровень 1: Задача (Workflow)

**Что видит пользователь:**
- Клик на ноду Roadmap → внутри Workflow DAG (Scout → Architect → Coder → Verifier).
- Команда подобрана автоматически (Dragon Silver/Bronze/Gold).
- Файлы определены (Scout уже нашёл нужные).
- Workflow template выбран (BMAD, quick_fix, refactor...).

**3 действия:**
1. **Запустить** (как есть, Архитектор всё подготовил)
2. **Изменить** (команду, workflow, описание)
3. **Назад** (к Roadmap)

**Автоматика:**
- Команда выбирается по сложности задачи (простое → Bronze, сложное → Gold).
- Workflow template выбирается по типу задачи (баг → quick_fix, фича → BMAD).
- Файлы определяются Scout'ом на основе описания задачи.
- Всё это происходит ДО того как пользователь зашёл в задачу.

---

## Уровень 2: Исполнение

**Что видит пользователь:**
- Тот же Workflow DAG, но ноды оживают: текущая светится, завершённые зелёные, будущие серые.
- Внизу — лог в реальном времени (что делает агент прямо сейчас).
- Таймер, токены, прогресс.

**3 действия:**
1. **Пауза**
2. **Отмена**
3. **Назад** (к Roadmap, pipeline продолжает в фоне)

---

## Уровень 3: Результат

**Что видит пользователь:**
- Задача завершена. Статистика: время, токены, стоимость.
- Табы: Код | Diff | Лог.
- Вердикт Verifier'а.

**3 действия:**
1. **Принять** (apply код)
2. **Переделать** (retry с feedback)
3. **Назад** (к Roadmap)

---

## Мини-окна

Три мини-окна, доступные на ЛЮБОМ уровне. Каждое: compact (маленькое, в углу) или expanded (полный экран).

| Окно | Compact | Expanded |
|------|---------|----------|
| **Чат с Архитектором** | Одна строка ввода + последний ответ | Полный чат с историей |
| **Список задач** | 5 задач, статусы | Все задачи, фильтры, поиск |
| **Статистика** | 4 числа (runs, success%, duration, cost) | Графики, тренды, per-agent breakdown |

Мини-окна не дублируют основной экран. Они — быстрый доступ без выхода с текущего уровня.

---

## Архитектор = Командир

Архитектор — самая умная модель. Он не просто отвечает на вопросы. Он:

- **Строит Roadmap** из анализа кодовой базы.
- **Назначает команды** по сложности (Bronze/Silver/Gold).
- **Выбирает workflow** (BMAD, quick_fix, refactor).
- **Определяет файлы** (через Scout + semantic search).
- **Предлагает следующую задачу** (Captain: зависимости выполнены → рекомендация).
- **Оценивает результаты** (вместе с Verifier).

Пользователь подтверждает. Не настраивает — подтверждает.

## Playground

**Одно предложение:** Playground — это полная копия проекта для безопасных экспериментов агентов.

**Правила:**
- Один playground на проект. Vetka = один playground.
- Полная копия архитектуры (git worktree).
- Агенты свободно работают внутри — читают, пишут, тестируют.
- Размер ограничен. Система сигнализирует при превышении лимита.
- Пользователь указывает путь: локальный диск или удалённый сервер.
- Результат playground → review → merge в основной проект.

**НЕ:**
- Не рандомные имена (vibrant-wright, etc).
- Не множество playground'ов.
- Не непонятная связь с задачами.

| Элемент | Почему убрать | Куда |
|---------|---------------|------|
| WorkflowToolbar (New/Save/Load/Validate/Generate/Import/Export) | 8 кнопок, нужны редко | Popup по ⚙ или автоматически |

| Breadcrumb | Ок, но дублирует навигацию | Оставить, но убрать тулбары выше/ниже |
| CaptainBar | Рекомендация | Встроить в Roadmap (подсветка рекомендованной ноды) |
| OnboardingModal | Нужен для первого запуска | Оставить, но упростить текст |

## Что всё взаимосвязано

Нет ничего что существует без связи:

```
Roadmap DAG
  └── Нода = Задача
       ├── Workflow DAG (шаги)
       ├── Команда (Dragon Bronze/Silver/Gold)
       ├── Файлы (определены Scout'ом)
       ├── Статистика (время, токены, стоимость)
       ├── Результат (код, diff)
       └── Playground (где агенты работали)
```

Каждая сущность видна в DAG (визуально) и в списке (текстом). Нет скрытых настроек без визуализации. Нет визуализации без функции.

## Контрольный вопрос

Если бабушка откроет Mycelium, она увидит:
1. Граф с подсвеченной нодой и надписью "Рекомендую начать с этого".
2. Одну большую кнопку "Запустить".
3. Прогресс: зелёные ноды, лог внизу.
4. Результат: "Готово. Принять?"

Четыре шага. Ноль настроек. Все решения принял Архитектор.

Хочешь настроить? Можно. Но по умолчанию — всё работает само.
