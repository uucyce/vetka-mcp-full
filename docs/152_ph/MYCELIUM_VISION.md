## Одно предложение

Mycelium автоматически строит карту проекта, расставляет задачи, подбирает команды, выполняет работу — пользователь только подтверждает. Прогресс сохраняется между сессиями.

## Проблема сейчас

Открываешь MCC — видишь 20+ кнопок, 3 тулбара, фильтры на пол-экрана. Непонятно что нажать первым. Непонятно что с чем связано. Создатель системы не понимает как ей пользоваться — значит система сломана.

## Принцип

**Максимум 3 действия за раз.** Не 3 кнопки плюс навигация плюс настройки. А 3 действия — всё что видит пользователь как выбор. Остальное скрыто или происходит автоматически.

Каждое состояние ошибки тоже даёт 3 действия для recovery. Нет тупиков.

---

## Матрёшка: 5 состояний, один экран

Mycelium — это DAG-в-DAG. Один и тот же экран, разное содержимое:

```
FIRST RUN  → Onboarding: выбери проект
ROADMAP    → Граф задач проекта (корень внизу, новое вверху)
ЗАДАЧА     → Клик на ноду — внутри workflow DAG
ИСПОЛНЕНИЕ → Workflow запущен, ноды светятся
РЕЗУЛЬТАТ  → Работа завершена, код/diff/лог
```

Переход: клик на ноду = вход внутрь, кнопка "назад" = выход наружу.

## First Run (Onboarding)

Первый запуск — нет проекта, нет Roadmap. Нужно начать.

**Что видит пользователь:**
- Чистый экран с одним вопросом: "Какой проект развиваем?"

**3 действия:**
1. **Выбрать папку** (локальный проект)
2. **Указать URL** (git clone)
3. **Пропустить** (описать проект текстом)

**Что происходит после:**
- Архитектор Проекта сканирует проект (Scout → semantic index → анализ структуры).
- Автоматически строит Roadmap.
- Создаёт Playground (копию для безопасной работы).
- Сохраняет конфиг в `project_config.json` — при следующем запуске всё на месте.

**Если ошибка:**
- Путь не существует → "Проверьте путь. Попробовать другой?"
- Git clone failed → "Нет доступа. Указать токен?"
- Диска мало → "Нужно N ГБ. Выбрать другой путь?"

## Roadmap

**Что видит пользователь:**
- Граф. Ноды = задачи. Стрелки = зависимости. Корень внизу.
- Подсветка: готово (зелёное), рекомендованное (яркое), заблокировано (тусклое).
- На рекомендованной ноде — текст Архитектора Проекта: "Начни с этого: [причина]".

**3 действия:**
1. **Запустить** (рекомендованную задачу)
2. **Спросить Архитектора** (открыть чат)
3. **Добавить задачу** (ручной ввод)

**Откуда берётся Roadmap:**
- Архитектор Проекта анализирует кодовую базу и строит граф автоматически.
- Пользователь может поправить (перетащить, удалить, добавить).
- Roadmap сохраняется на диск, обновляется по мере выполнения задач.

**Если ошибка:**
- Roadmap пустой (нет задач) → Архитектор: "Опишите что нужно сделать" (чат).
- Все задачи заблокированы (circular deps) → Подсветка проблемных стрелок, "Удалить зависимость?"
- API ключ недействителен → "Архитектор недоступен. Добавьте задачу вручную."

## Два Архитектора

В системе два уровня архитектора — это разные роли, разные модели, разные задачи:

### Архитектор Проекта (Project Architect)
- **Кто:** Самая умная доступная модель (Opus, GPT-5, Kimi K2.5).
- **Когда:** Работает на уровне Roadmap. Вне pipeline.
- **Что делает:**
  - Строит и обновляет Roadmap (граф задач).
  - Рекомендует следующую задачу (Captain).
  - Назначает команду и workflow template для каждой задачи.
  - Общается с пользователем в чате.
  - Оценивает результаты на уровне проекта ("эта фича закрывает issue X").
- **Аналогия:** Главный архитектор стройки. Смотрит на чертёж целиком.

### Архитектор Задачи (Task Architect)
- **Кто:** Модель из Dragon-команды (Kimi K2.5 / Qwen3-30b, зависит от тира).
- **Когда:** Работает внутри pipeline, на уровне Задачи.
- **Что делает:**
  - Получает описание задачи от Архитектора Проекта.
  - Разбивает на подзадачи (subtasks) — конкретные файлы, конкретные изменения.
  - Планирует порядок исполнения.
  - Делает re-plan если Verifier отклонил (escalation).
- **Аналогия:** Бригадир на участке. Знает что делать прямо сейчас.

### Связь между ними

```
Архитектор Проекта (Roadmap, стратегия)
  │
  │  "Задача: добавить dark mode. Файлы: App.tsx, useStore.ts.
  │   Команда: Dragon Silver. Workflow: BMAD."
  │
  ▼
Архитектор Задачи (Pipeline, тактика)
  │
  │  "3 подзадачи: 1) theme context, 2) toggle component, 3) CSS variables"
  │
  ▼
Scout → Researcher → Coder → Verifier
```

## Pipeline Workflow (полный)

Внутри каждой задачи работает проверенный pipeline. Все наработки сохранены:

```
Scout (префетч)
  │  ripgrep + semantic search + marker scan
  │  Результат: файлы, паттерны, маркеры
  ▼
Architect (план)                    Researcher (веб)
  │  Разбивает на подзадачи    ←→    │  Tavily + Context7
  │  marker_map для рельсов          │  Документация, API patterns
  │  PM pass (уточнение)             │  known_issues, web_sources
  ▼                                  ▼
Coder (FC loop, до 4 итераций)
  │  Function Calling: vetka_read_file, search_semantic, search_code, list_files
  │  Auto-read после search (top file)
  │  Marker rails: 📍 MARKER_SCOUT_1:42 [INSERT_AFTER]
  │  Patch mode для существующих файлов
  │  Project-aware: знает стек (Zustand, not MobX)
  ▼
Verifier (3-point check)
  │  HAS CODE / CORRECT / COMPLETE
  │  confidence gate < 0.75 → auto-fail
  ▼
  ├── Pass → Результат
  ├── Minor → Retry Coder (до 2 раз) с фидбеком
  └── Major → Escalate → Architect re-plan
              └── Tier upgrade (Bronze → Silver → Gold) при повторных неудачах
```

**Что уже работает и не трогаем:**
- FC loop (Coder вызывает инструменты во время генерации кода)
- Auto-read top file после search
- Marker rails (Scout → Architect → Coder)
- Parallel recon (Scout + Researcher параллельно)
- Verifier retry/escalation loop
- Tier upgrade при неудачах
- Per-agent stats collection (время, токены, retries)
- Project awareness (детектит стек проекта)
- STM (Short-Term Memory) между подзадачами
- Token limits per model
- Pipeline cancellation

## Задача (Workflow)

**Что видит пользователь:**
- Клик на ноду Roadmap → внутри Workflow DAG.
- Полный pipeline визуально: Scout → Researcher → Architect → Coder → Verifier.
- Команда подобрана автоматически (Dragon Silver/Bronze/Gold).
- Файлы определены (Scout уже нашёл нужные).
- Workflow template выбран (BMAD, quick_fix, refactor...).

**3 действия:**
1. **Запустить** (как есть — Архитектор всё подготовил)
2. **Изменить** (команду, workflow, описание — popup)
3. **Назад** (к Roadmap)

**Автоматика (происходит ДО того как пользователь зашёл):**
- Команда — по сложности (простое → Bronze, сложное → Gold).
- Workflow template — по типу (баг → quick_fix, фича → BMAD).
- Файлы — Scout + semantic search.

**Если ошибка:**
- Scout не нашёл файлов → "Укажите файлы вручную или опишите подробнее".
- Workflow template не подходит → "Изменить" → выбор из списка.
- Зависимости не выполнены (пришёл по прямому URL) → "Сначала выполните: [задача]" + кнопка перехода.

## Исполнение

**Что видит пользователь:**
- Тот же Workflow DAG, но ноды оживают: текущая пульсирует, завершённые зелёные, будущие серые.
- Внизу — лог в реальном времени.
- Таймер, токены, прогресс.

**3 действия:**
1. **Пауза**
2. **Отмена**
3. **Назад** (к Roadmap — pipeline продолжает в фоне)

**Если ошибка:**
- Агент завис (timeout) → Автопауза + "Повторить шаг?" / "Пропустить?" / "Отменить"
- Токены превысили бюджет → Автопауза + "Продолжить? (дорого)" / "Переключить на Bronze?" / "Отменить"
- Verifier отклонил → Автоматический retry (до 2 раз), потом → Результат с пометкой "Требует внимания"

---

## Результат

**Что видит пользователь:**
- Задача завершена. Краткая статистика: время, токены, стоимость.
- Табы: Код | Diff | Лог.
- Вердикт Verifier'а.

**3 действия:**
1. **Принять** (apply код → нода в Roadmap зеленеет)
2. **Переделать** (retry с feedback — поле ввода: "Что не так?")
3. **Назад** (к Roadmap, результат сохраняется — можно вернуться)

**Если ошибка:**
- Apply конфликтует (git merge) → Показать конфликт, "Решить вручную?" / "Отменить apply"
- Результат пустой (coder не выдал код) → "Переделать" с фидбеком уже заполненным: "Coder не произвёл результат"

## Мини-окна

Три мини-окна, доступные на ЛЮБОМ уровне. Каждое: compact (маленькое, в углу) или expanded (полный экран, overlay).

| Окно | Compact | Expanded |
|------|---------|----------|
| **Чат с Архитектором** | Одна строка ввода + последний ответ | Полный чат с историей |
| **Список задач** | 5 задач, статусы | Все задачи, фильтры, поиск |
| **Статистика** | 4 числа (runs, success%, duration, cost) | Графики, тренды, per-agent |

Expanded = overlay поверх основного экрана. Закрыл — вернулся на тот же уровень, ничего не потерялось.

## Статистика (→ Phase 155)

Статистика — полноценная подсистема. Не просто числа, а аналитика для принятия решений:

- **Per-agent breakdown:** кто сколько токенов потратил, кто чаще фейлит.
- **Team comparison:** Dragon Silver vs Bronze vs Gold — success rate, скорость, стоимость.
- **Trends:** улучшается ли pipeline со временем.
- **Cost report:** сколько стоит каждый прогон.
- **Weak links:** какой агент тормозит pipeline.

Backend уже готов (Phase 152: `pipeline_analytics.py`, 9 endpoints). Frontend (`StatsDashboard.tsx`) подключён, но требует доработки.

Полная реализация статистики связана с **Phase 155** (Unified Favorites + Model Weight Classes):
- Weight classes позволяют сравнивать модели по категориям, а не поимённо.
- Favorites дают Архитектору данные о предпочтениях пользователя.
- Eval agent использует статистику для автоматического улучшения команд.

→ См. `docs/155_ph/PHASE_155_UNIFIED_FAVORITES_AND_MODEL_WEIGHT_CLASSES.md`

## Архитектор Проекта = Командир

Архитектор Проекта — самая умная модель. Он не просто отвечает на вопросы:

- **Строит Roadmap** из анализа кодовой базы.
- **Назначает команды** по сложности (Bronze/Silver/Gold). → Phase 155: по weight classes.
- **Выбирает workflow** (BMAD, quick_fix, refactor).
- **Определяет файлы** (через Scout + semantic search).
- **Рекомендует следующую задачу** (Captain: зависимости выполнены → рекомендация).
- **Оценивает результаты** (вместе с Verifier).
- **Учитывает статистику** (→ Phase 155: weak links, team performance, user favorites).

Пользователь подтверждает. Не настраивает — подтверждает.

Хочешь настроить? Кликни "Изменить" на любом уровне — всё доступно. Но по умолчанию Архитектор решает сам.

## Playground

**Одно предложение:** Playground — полная копия проекта для безопасных экспериментов агентов.

**Правила:**
- Один playground на проект. Не несколько — один.
- Создаётся автоматически при onboarding (или вручную).
- Полная копия (git worktree). Агенты читают, пишут, тестируют внутри.
- Размер ограничен. Пользователь задаёт лимит. Система сигнализирует при превышении.
- Путь: локальный диск (по умолчанию) или адрес удалённого сервера.
- Результат: playground → review в "Результат" → merge в основной проект.
- Имя = имя проекта + "-playground" (не рандомные хеши).

**Если ошибка:**
- Диск полон → "Лимит N ГБ превышен. Очистить старые результаты?" / "Указать другой путь?"
- Git worktree failed → "Ошибка копирования. Попробовать заново?" / "Создать без git?"

## Что всё взаимосвязано

```
Проект (project_config.json)
  └── Playground (копия для агентов)
  └── Roadmap DAG (граф задач)
       └── Нода = Задача
            ├── Workflow DAG (Scout → Researcher → Architect → Coder → Verifier)
            ├── Команда (Bronze/Silver/Gold — подобрана автоматически)
            ├── Файлы (определены Scout'ом)
            ├── Результат (код, diff)
            └── Статистика (время, токены, стоимость) → Phase 155: weight classes
```

## Persistence

Всё сохраняется между сессиями:
- `project_config.json` — путь к проекту, playground, лимиты
- `roadmap.json` — граф задач с состояниями
- `task_board.json` — задачи, результаты, статистика
- Workflow'ы — сохраняются автоматически при изменении

Закрыл → открыл → всё на месте. Никакого "начни сначала".

## Контрольный вопрос

Бабушка открывает Mycelium:
1. "Какой проект?" → Выбирает папку.
2. Граф с яркой нодой. "Запустить?" → Да.
3. Ноды зеленеют одна за другой. Лог внизу.
4. "Готово. Принять?" → Да.

Четыре шага. Ноль настроек. Все решения принял Архитектор.

Если что-то пошло не так — 3 варианта recovery. Никогда не тупик.
