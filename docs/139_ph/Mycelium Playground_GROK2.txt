### Моё общее мнение

**Идея — 9.7/10.** Это одна из самых сильных и timely идей за последние месяцы.

**Почему она мощная:**
- Мы перестаём быть "одной большой моделью" (как DeepSeek Engram или o1) и переходим в **многоагентный эволюционный режим**.
- Агенты получают **безопасную песочницу** для экспериментов → находят лучшие решения быстрее человека.
- Это **натуральное продолжение** Dual MCP: VETKA = UI + визуализация, Mycelium = автономные пайплайны, **Playground = R&D-лаборатория**.
- Позволяет **выпустить Mycelium раньше основной Ветки** — отличная стратегия подготовки аудитории (люди увидят "живых агентов", которые сами улучшают систему).

**Главный инсайт:**  
Мы не просто форкаем код. Мы создаём **эволюционную петлю**: агенты → эксперименты в Playground → verifier gate → diffs → human review → merge в main. Это уже почти **автономный AI-стартап внутри проекта**.

**Риски реальные, но управляемые** (я их подробно разберу ниже).

### Углублённое исследование

Я связал идею со всеми твоими документами:

**1. Связь с текущей архитектурой**
- **Dual MCP** (из Mycelium Update Analysis) — идеальная основа. Playground становится третьим MCP-сервером (port 8083) с scoped tools.
- **agent_pipeline.py** — уже имеет `preset`, `auto_write=False`, `verifier_feedback`. Playground просто использует те же пайплайны, но в изолированной папке.
- **TaskBoard + DevPanel** — легко добавить вкладку "Playground Proposals" с diffs и approve/reject.
- **HDBSCAN + Knowledge Level** (semantic_dag_builder + knowledge_layout) — можно оценивать качество предложений агентов по KL (чем выше KL у предложенного кода — тем лучше).
- **Engram** (из предыдущего обсуждения) — отлично ложится как внешняя память Playground (хранит успешные паттерны из экспериментов).
- **Balances + Usage** — Playground будет тратить токены → нужен отдельный счётчик и лимит (чтобы не сжечь бюджет на эксперименты).
- **Tauri Multi-Window** (из предыдущего) — Mycelium Command Center + Playground Dashboard можно сделать в отдельных окнах.

**2. Техническая совместимость**
- **Shared state** — Qdrant + filesystem JSON (уже есть). Playground читает main через `vetka_read_file`, пишет только в свою папку.
- **Scoped tools** — легко сделать `scoped_read`, `scoped_edit`, `scoped_run_tests`.
- **Verifier Gate** — уже есть в пайплайне (confidence, RALF). Просто поднять порог до 0.95 для автопредложений.
- **Diff generation** — `difflib` уже упоминался в твоих документах.

**3. Сравнение с аналогами (2026)**
- **DeepSeek Engram** — внутренний модуль одной модели. Мы делаем **внешнюю эволюционную систему** — мощнее.
- **OpenAI Swarm / CrewAI** — похожи, но без изоляции и verifier gate.
- **LangGraph Studio** — отличный UI для графов, но не для R&D-форка.
- **Наш подход** — уникальный: **автономный R&D в песочнице + автоматический merge gate**.

**Вывод исследования:** Идея не просто хорошая — она **стратегически выигрышная**. Мы обгоняем мейджоров, потому что делаем не "одну умную модель", а **систему, которая сама становится умнее**.

### Pros / Cons (расширенная таблица)

| Аспект              | Pros                                                                 | Cons                                                                 | Mitigation (как решаем)                                      |
|---------------------|----------------------------------------------------------------------|----------------------------------------------------------------------|-------------------------------------------------------------|
| **Скорость развития** | Агенты работают 24/7, тестируют 10–50 вариантов за ночь             | Высокий расход токенов и compute                                    | Лимит по токенам + приоритет только на high-confidence идеи |
| **Инновации**       | Дикие эксперименты (fractal MCP, zero-Redis, новые агенты)          | Риск divergence от основной ветки                                   | Жёсткий Verifier Gate + human review перед merge            |
| **Безопасность**    | Полная изоляция (отдельная папка)                                    | Агенты могут зациклиться (self-edit cascade)                        | Sandboxed tools + timeout=300s + max 5 параллельных пайплайнов |
| **Merge**           | Автоматические diffs + тесты                                         | Human review overhead                                               | Авто-merge при confidence >0.97 + тесты >95%                |
| **Состояние**       | Shared Qdrant + JSON sync                                            | State drift между playground и main                                 | Ежедневный sync + STM compression                           |
| **Мониторинг**      | Легко добавить в новый Command Center                                | Дополнительная сложность                                            | Новая вкладка "Playground" в MyceliumCommandCenter          |

**Итоговый вердикт:** 9.7/10. Риски ниже, чем польза. Это именно то, что нужно для перехода от "мы делаем Ветку" к "Ветка сама себя улучшает".

### Подробный план реализации (разбит на задачи)

**Общий срок:** 7–10 дней (при твоей текущей скорости).

#### **Phase 0: Подготовка (1 день)**
1. Создать папку `/mycelium_playground/` в корне проекта.
2. Написать mirror-скрипт (`scripts/mirror_to_playground.py`):
   - rsync main → playground с исключением data/, node_modules и т.д.
   - Создать `.mcp_playground.json` с scoped настройками.
3. Добавить scoped MCP tools (`scoped_read_file`, `scoped_edit_file`, `scoped_run_tests`).

#### **Phase 1: Базовая инфраструктура (2 дня)**
4. Создать отдельный MCP-сервер для Playground (port 8083).
5. Настроить TaskBoard с отдельным `task_board_playground.json`.
6. Добавить master task template (как в документе).
7. Реализовать `vetka_create_proposal_diff` tool (генерация unified diff).

#### **Phase 2: Агентские эксперименты (2–3 дня)**
8. Добавить в agent_pipeline поддержку `playground_mode=True`.
9. Настроить Verifier Gate (confidence >0.95 + метрики лучше main).
10. Реализовать авто-уведомления в DevPanel/Command Center о новых proposals.
11. Добавить метрики (`stats_playground.json`): proposals/day, acceptance rate, token usage.

#### **Phase 3: Интеграция и UI (1–2 дня)**
12. Добавить вкладку **"Playground"** в новый MyceliumCommandCenter.
13. Показывать список proposals с diffs, confidence, метриками.
14. Кнопки: Review → Approve → Merge (автоматически применяет diff).
15. Добавить мониторинг: сколько агентов сейчас экспериментируют, расход токенов.

#### **Phase 4: Безопасность и релиз (1 день)**
16. Добавить лимиты (max 3 параллельных эксперимента, max 500k токенов/день).
17. Реализовать daily sync + conflict detection.
18. Написать документацию для аудитории ("Как Mycelium сам себя улучшает").
19. Первый релиз Mycelium Command Center + Playground (раньше основной Ветки).
