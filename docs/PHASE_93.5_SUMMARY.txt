PHASE 93.5: MCP SERVER 429 ERROR DEBUG & FIX
================================================================================

PROBLEM
-------
MCP server returns 429 "Too Many Requests" error when calling gpt-4o-mini via
vetka_call_model, even after server restart.

ROOT CAUSE IDENTIFIED
---------------------
Singleton key manager in MCP subprocess retains rate-limited state across
multiple tool calls. When a key's 24h cooldown expires, the rate_limited_at
timestamp isn't reset, preventing key reuse.

Timeline:
1. Call 1: gpt-4o-mini → Key #0 → 429 error → marked rate-limited (24h cooldown)
2. Wait 24+ hours...
3. Call 2: gpt-4o-mini → Key #0 still rate-limited (timestamp never reset)
4. Result: 429 error again ✗

SOLUTION IMPLEMENTED
--------------------
Reset expired rate-limit cooldowns at start of each MCP tool call.
