2. Approach — Step‑by‑Step Plan

The core issue is that the Stage block contains a preview‑mode switcher with a depth option, making it a duplicate surface for depth‑map visualisation that competes with the Depth block. The intended mental model (from PARALLAX_PRODUCT_UI_RESET_PROJECT_2026-03-29.md) is:

Stage → final parallax composite preview only.

Depth → dedicated depth‑map tool with its own B/W preview (Resolve‑like).

Step 1 – Remove depth preview mode from Stage

Locate the PreviewMode type definition in App.tsx (likely type PreviewMode = 'composite' | 'depth' | 'selection').

Change it to 'composite' | 'selection'.

Find the segmented control JSX inside the Stage block and delete the button that switches to 'depth'.

Update any useState default value for preview mode to 'composite'.

Remove any conditional rendering that checks previewMode === 'depth' (replace with composite or remove the branch).

Step 2 – Add a dedicated depth preview inside the Depth block

To give the Depth block its own “truth surface”, we will create a small canvas that renders the current depth map independently.

Extract depth image data
The existing WebGL renderer (or canvas 2D pipeline) already knows how to draw the depth map (used previously in Stage’s depth mode). We need to expose a method to obtain the depth map as an ImageData or an HTMLCanvasElement.
If not available, add a function in the engine (e.g., getDepthMapImageData()) that reads the depth texture or recomputes the depth visualisation.

Build a DepthMapPreview component inside App.tsx

Use a useRef for a <canvas> element.

In a useEffect, set up an interval or subscribe to depth‑parameter changes (near/far/gamma, etc.) and to frame updates.

On each update, call the engine’s depth‑image getter and draw it onto the canvas.

Place the preview in the Depth block, above the sliders or as a prominent card.

Step 3 – Adjust UI layout and styling for clarity

Stage block – rename heading to “Composite Preview” (optional) and reduce its vertical padding if it feels “giant”.

Depth block – give the new depth preview canvas a consistent size (e.g., 100% width, 180px height) with a clear label “Depth Map Preview”.

CSS changes – ensure both blocks have distinct visual roles; maybe add a subtle border or background to the depth preview to emphasise it is a different tool.

Step 4 – Verify and clean up

Remove any leftover references to 'depth' preview mode in event handlers or props.

Run npm run build to confirm no TypeScript errors.

Test interactively: changing near/far/gamma should update the depth preview; Stage should always show the final composite.

3.2 App.tsx – depth preview inside Depth block

Add a helper to access depth image data.

tsx

Download
const DepthMapPreview = () => {
  const canvasRef = useRef<HTMLCanvasElement>(null);

  const updatePreview = useCallback(() => {
    const canvas = canvasRef.current;
    if (!canvas) return;
    const depthImage = (window as any).vetkaParallaxLab?.getDepthMapImageData?.();
    if (depthImage && depthImage instanceof ImageData) {
      canvas.width = depthImage.width;
      canvas.height = depthImage.height;
      const ctx = canvas.getContext('2d');
      ctx?.putImageData(depthImage, 0, 0);
    }
  }, []);

  useEffect(() => {
    // Initial update
    updatePreview();
    // Subscribe to depth parameter changes and frame redraws
    const interval = setInterval(updatePreview, 100); // simple polling – replace with proper event
    return () => clearInterval(interval);
  }, [updatePreview]);

  return <canvas ref={canvasRef} className="depth-preview-canvas" />;
};

4. Risks — Edge Cases
Risk	Mitigation
The engine does not expose getDepthMapImageData().	Extend the renderer module (e.g., lib/parallaxRenderer.ts) to add a method that returns the current depth visualisation as ImageData. Reuse the shader/composite logic that previously drew depth mode.
Performance degradation due to polling the depth image every 100 ms.	Replace polling with an event‑based subscription: emit an event whenever depth parameters change or a new frame is rendered. The depth preview updates only on those events.
The depth preview canvas size mismatches or looks pixelated.	Use imageSmoothingEnabled = false on the canvas context to keep a sharp pixel‑accurate look (depth maps are often low‑res). Alternatively, match the canvas resolution to the source image size.
Selection mode in Stage still refers to a visualisation that may depend on depth data.	This is acceptable – selection mask does not duplicate depth behaviour. No change needed.
The Stage canvas might internally still support a depth mode that is now unreachable via UI.	That is harmless; we can keep the internal capability for debugging, but the UI no longer exposes it.

index.css – only style additions; no breaking changes.
