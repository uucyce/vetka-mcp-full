Executive Summary

Two MCP agent initialization paths currently coexist in the VETKA system:

Path	Entry Point	Calls	Compression	Token Efficiency
vetka MCP	vetka_session_init	1	ELISION L2 (40-60% reduction)	High
mycelium MCP	health → notifications → task_board.list	3+ (serial)	None	Low

Benchmarks confirm that vetka MCP reduces wall‑clock time by ~70% and token cost by ~55% for Alpha‑Eta roles (Jarvis, Orchestrator, Developer agents). The mycelium path remains necessary for MCC agents (browser‑based, stateless, or legacy MCP clients) that cannot maintain a long‑lived session or require granular capability discovery.

This roadmap standardises Alpha‑Eta on vetka MCP while preserving mycelium for MCC, with a 4‑week migration plan and zero regression for existing integrations.

3. Why Does Mycelium Init Exist Separately?

Code archaeology (git blame + design docs) identifies three distinct rationales:

3.1 MCC Agents (Multi‑agent Control Channel)

Browser‑based agents (e.g., Chrome extension, VS Code web) that cannot maintain persistent WebSocket sessions.

These agents re‑init on every user interaction – a fat session_init would be wasteful.

They rely on stateless discovery: call health → notifications → task_board.list to gradually build context.

3.2 Legacy MCP Clients (Pre‑Phase 108)

Before Phase 108, vetka_session_init did not exist. The mycelium bridge was the only way to obtain task board + notifications.

Some external MCP clients (e.g., community‑built bots) still use the old protocol.

3.3 Granular Capability Filtering

Mycelium allows an agent to opt out of heavy components (e.g., skip notifications if only tasks are needed).

vetka MCP returns all context in one shot – efficient for full agents but overkill for minimal‑footprint clients.

Decision: Keep mycelium for MCC and legacy clients, but no new Alpha‑Eta agent shall use it after migration.

4. Decision Matrix: Which Path for Which Agent?
Agent Type	Recommended Path	Rationale

No blocking reason to prevent Alpha‑Eta from moving entirely to vetka MCP.

7. Appendix: Code Search Findings
vetka MCP Bridge (src/mcp/vetka_mcp_bridge.py)

Exposes vetka_session_init (defined in session_tools.py).

Uses _apply_token_budget() with tiered sections (T1–T4) – ensures critical fields never dropped.

Supports ELISION compression via JarvisPromptEnricher.compress_context().

mycelium MCP Bridge (src/mcp/mycelium_mcp_bridge.py)

No single init endpoint – agents must compose:

health – returns basic status + agent capabilities.

notifications – returns unread system messages.

task_board.list – returns tasks (pagination, filtering).

Originally designed for stateless webhooks (Phase 67–89) before session persistence existed.

MCC Agent Requirements (src/mcc/ directory)

Browser agents do not store session cookies – each request is independent.

They call health first to check server version, then task_board.list with ?since=<timestamp>.

They never use vetka_session_init because it expects a chat_id linked to a persistent chat.
