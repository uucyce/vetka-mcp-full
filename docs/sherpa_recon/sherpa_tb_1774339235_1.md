Research Task: [DEBRIEF-BUG] STM Load Decay Reset Issue
1. Problem Statement

The Short‑Term Memory (STM) module currently loads persisted memory entries via from_dict(), preserving original last_accessed timestamps and access counts. This causes two problems in a new session:

Decay does not reset – memories that were old in the previous session immediately become “cold” in the new session, even though they are being reloaded as fresh context.

Session boundary ignored – a new session should treat all loaded memories as equally relevant at start, without artificial aging from past usage.

Observed symptom
Important memories from prior sessions are prematurely evicted or given low relevance scores because their decay weight is computed from stale timestamps.

2. Root Cause Analysis

The STM loader (likely in src/memory/stm.py or similar) uses a generic from_dict factory that copies every field verbatim:

class STMEntry:

        return STMEntry(
            content=data["content"],
            last_accessed=data["last_accessed"],   # ← old timestamp
            access_count=data["access_count"],

No distinction is made between:

Deserialising for internal migration / backup (preserve timestamps)

Loading into a live session (reset timestamps to simulate fresh access)

Because the same code path is used for both, a new session inherits the decay state of the old one.

3. Proposed Fix – Session Boundary Weight Adjustment

Introduce a session_boundary flag in the load method that resets the temporal decay factors.

3.1 High‑Level Design

Add a reset_decay_on_load: bool = True parameter to the STM load function.

When True, set last_accessed = current_time and optionally access_count = 1 (or retain count but adjust weight formula).

Keep original timestamps only for debugging or audit (e.g., store an original_timestamp field).

class ShortTermMemory:
    def load_from_dict(self, data: dict, reset_decay: bool = True) -> None:
        for entry_dict in data.get("entries", []):
