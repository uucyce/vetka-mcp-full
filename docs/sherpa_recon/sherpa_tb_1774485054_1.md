Research Task: File Picker Returns Only Filename (No Full Path in Browser Context)
Issue Summary

Bug ID: DEBRIEF-BUG-20260408-001

Title: File picker in browser context returns only filename, not full path – needs Electron/Tauri native dialog for production

Affected Component: Frontend file upload / file picker (React)

Observed Behavior:
When running in a pure browser context (no Electron wrapper), input type="file" returns a File object whose webkitRelativePath or path property is undefined or just the basename. The current fallback (file as any).path works only inside Electron’s webUtils.getPathForFile or similar native integration.

Impact:

Cannot resolve absolute file paths for FFmpeg processing in the backend.

Breaks video file referencing, thumbnail generation, and timeline imports.

Production build (standalone web app) fails to access user-selected files correctly.

Root Cause Analysis

Browser security model – Modern browsers deliberately hide full filesystem paths from JavaScript for privacy/security reasons. Only the filename, size, MIME type, and last modified timestamp are exposed.

Electron-specific API – Electron provides webUtils.getPathForFile(file) which can return the real path if the file was dragged from the filesystem or selected via native dialog. This is not available in a standard web context.

Proposed Solutions
Solution A: Electron Native Dialog (Production Desktop)

Use @electron/remote or dialog.showOpenDialog to get full paths.

Implementation:

// Preload script exposes:
window.electronAPI.openFileDialog = () => ipcRenderer.invoke('dialog:openFile');

Pros: Full path, works with any file.

Cons: Only works in Electron wrapper – not for pure web deployment.

Solution B: Tauri Native Dialog (Alternative Desktop)

If migrating to Tauri, use tauri:fs or dialog module.

Example:

Pros: Cross‑platform, well‑typed, async.

Cons: Requires Tauri architecture (not React-only).

Solution C: Workaround for Browser Context – Read File as Blob

Do not require full path. Instead, read file contents as ArrayBuffer/Blob and send binary data to backend.

Backend receives the binary stream and saves to a temporary file with a generated name.

FFmpeg command references the temporary file path created on server.

Implementation:

const file = e.target.files[0];
const arrayBuffer = await file.arrayBuffer();

Pros: Works in any browser, no path needed.

Cons: Additional server‑side storage, larger memory footprint for large videos (but streaming via ReadableStream is possible).

Solution D: Drag‑and‑Drop with DataTransferItem (Limited)

For drag‑and‑drop, the DataTransferItem may expose a path on Chromium if the file is dragged from the filesystem (item.webkitGetAsEntry()). Complex and unreliable across browsers.

Recommended Approach for VETKA

Given VETKA is a video NLE with large files, Solution C (Blob upload + server temp file) is the most robust for a web deployment. For the desktop Electron/Tauri builds, implement Solution A or B as a native fallback that returns the full path directly, avoiding unnecessary file transfer.

async function getFilePathOrUpload(file: File): Promise<string> {
  // Try to get native path first (Electron/Tauri)
  if (window.electronAPI?.getPathForFile) {
    const nativePath = await window.electronAPI.getPathForFile(file);
    if (nativePath) return nativePath;
  }
  // Fallback: upload file contents and get server‑assigned path
  const uploadedPath = await uploadFileToServer(file);
  return uploadedPath;
