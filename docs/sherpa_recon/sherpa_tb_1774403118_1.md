🔍 Debrief: store_learning() Async/Sync Mismatch
Summary

The store_learning() function was implemented only as an async coroutine. However, the MCP complete handler (used for autocompletion requests in the Model Context Protocol server) runs in a synchronous context and needs to call store_learning() directly. Because the MCP server subprocess already has a running asyncio event loop, calling asyncio.run(store_learning(...)) or awaiting the coroutine inside a sync function is impossible without triggering RuntimeError: This event loop is already running.

Root Cause

store_learning() is defined as async def store_learning(...) in src/memory/learning_store.py (hypothetical location – actual file not provided but inferred from usage).

The MCP complete handler (likely in src/mcp/server.py or src/mcp/handlers.py) is a synchronous callback that must return a result quickly. It cannot be made async without breaking the MCP framework’s expected signature.

Attempts to call store_learning() directly from the sync handler fail because:

await store_learning(...) is invalid outside an async function.

asyncio.run(store_learning(...)) raises RuntimeError because an event loop is already running in the MCP subprocess.

Impact

Autocomplete functionality that depends on storing learning from user interactions is broken.

No learning entries are persisted when triggered via MCP complete calls, leading to missing memory for future sessions.

🛠️ Solution: Provide a Synchronous Wrapper

Implement a sync variant store_learning_sync() that safely schedules the async coroutine on the existing event loop without blocking or creating a new loop.

import asyncio
import logging
from typing import Any, Dict, Optional

logger = logging.getLogger(__name__)

# Original async function (kept as is)
async def store_learning(
    task_id: str,
    content: str,
    tags: Optional[list] = None,
    source: str = "mcp_complete",
) -> Dict[str, Any]:
    """Store a learning entry asynchronously."""
    # ... existing implementation ...
    pass


# NEW: Synchronous wrapper for MCP complete handler
def store_learning_sync(
    task_id: str,
    content: str,
    tags: Optional[list] = None,
    source: str = "mcp_complete",
) -> Dict[str, Any]:
    """
    Sync version of store_learning for use in MCP complete handlers.
    Schedules the async coroutine on the already-running event loop.
    Returns an empty dict immediately – learning is stored in background.
    """
    try:
        # Get the currently running event loop (MCP subprocess has one)
        loop = asyncio.get_running_loop()
    except RuntimeError:
        # No running loop – rare in MCP context, but fallback to asyncio.run
        logger.warning("No running event loop, using asyncio.run() for store_learning")
        return asyncio.run(store_learning(task_id, content, tags, source))

    # Schedule the coroutine as a background task on the existing loop
    # The sync handler does not wait for completion – learning is fire‑and‑forget.
    loop.create_task(store_learning(task_id, content, tags, source))
    logger.debug(f"Scheduled store_learning for task {task_id} on running loop")
    return {"status": "scheduled", "task_id": task_id}

📝 Additional Notes

Fire‑and‑forget: The sync wrapper does not wait for the async function to finish. If the MCP handler needs the result, we would need a more complex mechanism (e.g., asyncio.run_coroutine_threadsafe + future). But for learning storage, background execution is acceptable.

Error handling: Any exception inside store_learning is logged but does not affect the MCP response.

Alternative approach (if waiting is required): Use asyncio.run_coroutine_threadsafe() with a concurrent.futures.Future and block until done – but this risks deadlock if the loop is occupied. The fire‑and‑forget design is preferred for learning storage.
