Based on the search results and the task description, I can now provide a comprehensive solution for the Mycelium WebSocket handshake failure bug. The issue is that something is sending plain HTTP requests to the WebSocket endpoint on port 8082 instead of proper WebSocket upgrade requests.
Root Cause Analysis
The error websockets.exceptions.InvalidMessage: did not receive a valid HTTP request with EOFError: stream ends after 0 bytes typically occurs when:
Health check probes (like Kubernetes or Docker health checks) hitting the WS port with plain HTTP
Browser preflight requests or misconfigured clients
Load balancers/proxies sending TCP health checks without proper WebSocket headers 
The websockets library logs full tracebacks by default at ERROR level, which creates log noise .

async def _websocket_handler_with_graceful_errors(websocket, path):
    """
    Wrapped WebSocket handler that catches InvalidMessage gracefully.
    
    Prevents full tracebacks in logs when health checks or misconfigured
    clients send plain HTTP requests to the WebSocket endpoint.
    """
    global _invalid_http_count, _invalid_http_sources
    
    # Log remote address BEFORE handshake to identify WHO connects
    remote_addr = websocket.remote_address
    remote_str = f"{remote_addr[0]}:{remote_addr[1]}" if remote_addr else "unknown"
    
    try:
        # The handshake happens here - catch InvalidMessage specifically
        # This is where plain HTTP requests fail
        await websocket.wait_closed()
        
    except InvalidMessage as e:
        # Graceful handling of invalid HTTP requests (health checks, probes, etc.)
        _invalid_http_count += 1
        if remote_str not in _invalid_http_sources:

                f"(new source, total_invalid={_invalid_http_count}). "
                f"Likely health check probe or misconfigured client. "
                f"Connection rejected gracefully."

            # Log at DEBUG level for known sources to reduce noise
            logger.debug(
                f"[WS-BUG-8082] Invalid HTTP from known source {remote_str} "
                f"(count={_invalid_http_count})"
            )
        # Connection is already closed by websockets library, just return
        return
        
    except Exception as e:
        # Log other exceptions but don't spam full tracebacks
        logger.warning(f"[WS] Connection error from {remote_str}: {type(e).__name__}: {e}")

Key Changes Made
1. Pre-Handshake Connection Logging (process_request)
The process_request callback logs the remote address before the WebSocket handshake occurs . This identifies WHO is connecting even if the handshake fails.
2. Graceful InvalidMessage Handling
Wrapped the WebSocket handler to catch InvalidMessage specifically and log a clean one-liner instead of full tracebacks . The handler:
Tracks unique sources of invalid requests (_invalid_http_sources)
Logs new sources at WARNING level with full context
Logs repeat offenders at DEBUG level to reduce noise
Maintains a count for debugging purposes

The logging format [WS-BUG-8082] makes it easy to grep logs and identify:
Which IPs are sending invalid requests
