## Research Response

[DEBRIEF-BUG] embed_texts_for_overlay uses deterministic SHA1 fallback instead of semantic ranking

Severity: Medium
Affected Component: Video overlay / text embedding pipeline
Detected by: Agent Zeta (cross-cutting)
Project: VETKA (React + FastAPI + FFmpeg)

1. Bug Summary

The function embed_texts_for_overlay (location: likely src/video/overlay.py or src/embedding/text_embedder.py) currently implements a fallback mechanism when the JEPA runtime is not available. Instead of performing semantic ranking of text snippets for video overlays, it falls back to a deterministic SHA1 hash of the input text. This results in a ranking that is purely hash‑based, not semantically meaningful, and effectively random with respect to relevance.

Expected behaviour:
When the JEPA (or Ollama) semantic embedding service is unavailable, the system should either:

Gracefully degrade by using a simple keyword‑overlap or TF‑IDF fallback, or

Refuse to perform ranking and raise a clear error.

Actual behaviour:
The function silently uses hashlib.sha1(text.encode()).hexdigest() to sort or rank texts, producing a stable but semantically meaningless order.

2. Root Cause

The embedding pipeline was designed with the assumption that JEPA would always be running.

A “safe” deterministic fallback was added during development to avoid crashes, but no semantic fallback (e.g., local Ollama, on‑the‑fly TF‑IDF, or a lightweight sentence transformer) was implemented.

The SHA1 hash was chosen for reproducibility, not for semantic similarity.

No health check or startup logic ensures JEPA is running before embed_texts_for_overlay is called.

3. Impact

Video overlay relevance: Overlay text (e.g., keyword highlights, contextual captions) may appear in an order that does not match the user’s intent or the video’s content.

User experience: For features like “smart captions” or “contextual labels”, the ranking becomes essentially random, reducing the perceived intelligence of the NLE.

Debugging difficulty: The behaviour is deterministic per string, but not obviously wrong – the issue can go unnoticed for a long time.

Resource waste: If JEPA is intentionally disabled, the system continues to run but produces low‑quality output instead of failing fast or informing the user.

4. Proposed Solution

Replace the SHA1 hash fallback with a proper semantic fallback chain, and ensure that the system’s embedding service is reliably available.

High‑level approach:

Service health check – Before any ranking call, verify that JEPA (or configured embedding service) is reachable.

Fallback hierarchy:

Primary: JEPA HTTP service (semantic embeddings)

Secondary: Local Ollama instance (if available)

Tertiary: On‑the‑fly lightweight model (e.g., sentence-transformers/all-MiniLM-L6-v2 loaded once)

Last resort: Log an error and return texts in original order (or raise a clear exception)

Configuration – Allow the user/operator to specify the preferred embedding provider via config/embedding.yaml or environment variables.

Remove the SHA1 fallback completely – it provides no semantic value.

Alternative (simpler) solution:
Mandate that JEPA is always running. Add a systemd service or docker‑compose health check that starts JEPA automatically when the VETKA backend starts. If JEPA cannot be started, the backend fails fast with a clear error message.
