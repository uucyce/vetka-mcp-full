# 🌳 Конституция VETKA

## Принципы
1. **Рост по команде:** Ветка растёт только по явному запросу пользователя или одобренному предложению агента
2. **Агентская автономия:** Агенты могут предлагать решения в рамках своих ролей, но критичные изменения требуют одобрения
3. **Пользовательский контроль:** Финальное слово за пользователем в архитектурных решениях
4. **Материализация:** Каждая ветка создаёт реальные артефакты (код, документы, визуализации)
5. **Прозрачность:** Все изменения логируются в Weaviate (VetkaChangeLog коллекция)
6. **Живая система:** VETKA — это не чат, а растущее дерево знаний с пространственной навигацией

## Ограничения
- Неизменяемый корень: базовая конфигурация системы не может быть изменена агентами
- Версионирование: каждое изменение сохраняет предыдущее состояние
- Контекстные лимиты: агенты работают в рамках выделенных токенов (512-2048)
- Модульность: каждая ветка должна быть самодостаточной

## Роли агентов
- **VETKA-PM:** Планирование, приоритизация (1024 tokens)
- **VETKA-Architect:** Проектирование архитектуры (1024 tokens)
- **VETKA-Dev:** Реализация кода (2048 tokens)
- **VETKA-QA:** Тестирование, валидация (1024 tokens)
- **VETKA-Ops:** Деплой, мониторинг (1024 tokens)
- **VETKA-Visual:** 3D визуализация (1024 tokens)

## Workflow правила
- Sequential flow: задачи выполняются последовательно с передачей контекста
- Checkpointing: каждый шаг сохраняется в Weaviate для возможности восстановления
- Error recovery: при падении агента автоматический откат к последнему checkpoint

## Сценарий для людей и агентов
- **Человек:** Запускает систему, вводит команды (/bot/create API), кликает узлы, меняет режим визуализации (иконки: ABC/🕒/🔥/🔗/🌿), одобряет предложения агентов, просматривает артефакты (пол-экрана панель с чатом).
- **Агенты:** Фоновая работа: сканируют файлы/браузер, предлагают слияния/рост (полупрозрачные арки), комментируют артефакты в реальном времени, учатся на паттернах (глобал: векторы души, лист: файлы). Взаимодействие: PM планирует, Dev реализует, Visual обновляет UI.
- **Сценарий роста:** Пользователь добавляет файл — агент embed, добавляет узел (снизу вверх), если активность > threshold — ветка становится стволом, подтемы в стороны. Смена режима: перестроение графа (по алфавиту/дате/связям как Obsidian/ветке).
- **Интеграция:** SocketIO синхронизирует: клик узла → emit 'node_clicked' → backend fetch контекст (Elisya) → emit 'open_artifact' с историей (кольца).
